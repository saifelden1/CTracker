Snapshot — what is this app?

CTracker is a desktop app you run on your own computer to keep track of the things you're learning and building. Think of it as a personal dashboard for:

  • Courses you're taking (with their sub-units and lessons)
  • Projects you're working on (with priority, deadline, team, links)
  • To-do items for the day
  • Pomodoro timer sessions (the 25-minutes-on / 5-minutes-off study technique)
  • A calendar with per-day notes
  • Analytics — a GitHub-style heatmap of how active you've been

It's written in C++17 using Qt 6 (a toolkit for making desktop windows, buttons, charts, etc.) and stores all your data in a single small file on your disk using SQLite (a tiny database that needs no separate server).

About the "v1" / "v2" labels you'll see everywhere. They refer to two versions of the database layout — not two versions of the app.

• v1 = the original 4 tables (Courses/Projects, Units, Sessions/Tasks, Activity Log). That was the starting point.
• v2 = v1 plus six new tables (Categories, ProjectMeta, Todos, PomodoroSessions, CalendarDayDetails, Settings) and two new columns added to the existing CoursesProjects table.

A user who installed the app early on has a v1 database file; the app now ships with code ("migration") that quietly upgrades that file to v2 on first launch — without losing anything. So the codebase carries both labels at the same time: the v1 stuff is "the original parts", the v2 stuff is "what we added later".

Here's what is finished today:

  • Phase 0 — code organised neatly into folders, one per feature. phase-0-explained.md
  • Phase 1 & 2 — build system set up + the database (both v1 and v2 layouts, plus the upgrade code). phase-1-and-2-explained.md
  • Phase 3 — the C++ "shapes" (called structs) that describe one course, one todo, one pomodoro session, etc., so the rest of the app has a shared vocabulary. phase-3-explained.md
  • Phase 4 — the database talking layer: every read and write function the future UI needs (add a category, toggle a todo, save the pomodoro timer, …). phase-4-explained.md

The screens you can currently click around in are the older slice: a sidebar, a home page listing your courses and projects, a detail page for each one (with the progress slider that updates the heatmap), an analytics page with the heatmap, and a minimal settings page (just import / export). In addition, CoursesView (Task 7.3) is now built — a responsive grid of course/project cards with a filter bar and empty-state placeholder. The remaining pages (ProjectsView, TodoView, PomodoroView, updated HomeDashboard, updated AnalyticsView, updated SettingsView) are still coming in Phase 7.

Reading this diagram: The app is built in layers, like a cake. At the top is the window the user sees (shell). Below it are the individual screens (views). Below that, smaller reusable parts (widgets) like a circular progress bar or a heatmap. Below the widgets are models (helpers that prepare data for tables/lists). At the bottom is the data layer: one object called DatabaseManager that is the only thing in the app allowed to talk to the SQLite file.

Arrows: a solid arrow means "calls a function on"; a dashed arrow means "listens to a signal" (Qt's way of saying "hey, something changed, refresh yourself").

Colours: green = built today, amber = older version is built but more is coming, grey = not started yet.

Database schema — what's stored on disk

The file ctracker.db holds 10 tables (think of a table as a spreadsheet with named columns and rows). The diagram below shows each one and how they're connected.

The first 4 are the "original" (v1) tables and they already drive the UI you can click around in. The next 6 are the "added later" (v2) tables — they're created and seeded with defaults, the database functions to read/write them are all wired up (Phase 4), and they're now just waiting for the screens (Phases 5–9) that will actually use them.

A foreign key (FK) is a column that points at the ID of a row in another table — like saying "this Unit belongs to Course #5". When the parent row is deleted, two choices: either the child rows are deleted too (CASCADE) or the link is just set to nothing (SET NULL) and the child rows survive.

What the delete rules mean in plain words: if you delete a course, its units, sessions, and activity log entries also get deleted — they only existed to belong to that course. But your pomodoro sessions for that course are kept, with the link to the (now-gone) course set to "no course" — you still studied those minutes, the history shouldn't lie about it. Same idea for categories: deleting a category un-tags every course it labelled, but doesn't delete those courses.

What happens when you launch the app

Following main() step by step: Qt starts up, the database file is opened (or created if it's your first launch), the table layout is built or upgraded if needed, then the main window appears. The diagram shows each step as a message between parts of the app.

Why the "BEGIN TRANSACTION / COMMIT" wrap? A transaction is "all or nothing": if anything inside fails or the computer crashes mid-way, none of the changes are written and the database stays in its previous state. So if upgrading from v1 to v2 fails, you're not left with a half-upgraded file — you can just try again on next launch.

What happens when you drag a progress slider

This is the main thing you do in the app today. Inside a course, each lesson/session has a slider you drag from 0% to 100%. When you let go, the app does five things: save the new value, record an "activity log" entry (so the heatmap lights up that day), re-compute the overall progress ring on the course, refresh the home page so the card shows the new number, and re-paint the heatmap. The diagram shows the chain.

    STR->>UEW: sessionTaskProgressChanged(id, old=40, new=60)
    UEW->>EDV: onSessionProgressChanged(id, 40, 60)
    EDV->>DB:  updateSessionTaskProgress(id, 60)

    DB->>DB: getSessionTaskProgress(id) → 40 (oldValue)
    Note over DB: if old == new, return true (no-op guard)
    DB->>FS: BEGIN TRANSACTION
    DB->>FS: UPDATE SessionsTasks SET CurrentProgress=60
    DB->>FS: SELECT cp.Type via JOIN (determine "Course" / "Project")
    DB->>FS: INSERT INTO ActivityLog (...)
    DB->>FS: COMMIT
    DB-->>EDV: true
    DB->>HD:  dataChanged (signal, broadcast)
    DB->>ALM: dataChanged
    DB->>CH:  dataChanged (via ALM model)

    EDV->>EDV: refreshOverall() (recompute ring %)
    HD->>HD:  refreshCards() (every card re-pulls its progress)
    ALM->>ALM: refresh() (reload entries, rebuild caches)
    CH->>CH:  re-aggregate daily counts → repaint
The two key ideas here: (1) Transaction — saving the new slider value and writing the activity-log entry happen together; either both land or neither does. So the heatmap can never lie about something that didn't get saved. (2) Signals — instead of every screen having to know who to call when something changes, the database just shouts dataChanged into the air, and every interested screen listens and refreshes itself. Much less wiring.

Class map — what owns what

A class in C++ is a blueprint for an object. The diagram shows the classes that exist today and how they're related:

  • A line with a triangle (--|>) means "is a kind of" — e.g. MainWindow is a kind of QMainWindow, which is a kind of QWidget, which is a kind of Qt's base QObject. This is called inheritance.
  • A line with a filled diamond (*--) means "owns and contains" — when the owner is destroyed, so is the thing it owns.
  • A line with an empty diamond (o--) means "uses but doesn't own" — the lifetime is managed somewhere else.

About the design/ folder. You'll see a folder called design/ in the repo. It's a separate React + Tailwind prototype that was built as a visual mockup only. None of its code is included in the actual Qt app — we just read the JSX/Tailwind files to see colours and spacing, then re-implement the same look using Qt widgets and QSS. The Qt app never executes any React code.
